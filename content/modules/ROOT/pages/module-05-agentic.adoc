= System Administration with Agentic AI 

In the last module, you stepped into the developer role building applications using private LLM endpoints. Now, you'll become a **site reliability engineer (SRE) or DevOps practitioner** focused on observability, diagnostics, and operational scale.

But you won't go at it alone, you'll work with agentic AI tooling to navigate your OpenShift environment in real time - just like a modern SRE team building insight-driven automation.

NOTE: You're still using the **same Nemotron model** from the MaaS deployment you accessed in earlier modules. The difference? Instead of consuming it through Continue for coding assistance, you'll now interact with it through the Gen AI Studio Playground with MCP server integrations for infrastructure operations. This demonstrates how a single, centrally-managed model serves multiple personas and use cases across your organization.

== Why Agentic AI for Admins?

Think of agentic AI as your copilot for cluster operations - ready to answer complex infrastructure questions using reasoning and real-time data.

Instead of navigating complex dashboards, YAML, and shell scripts, you'll try something new:

* “What’s consuming the most memory in our dev cluster right now?”

* “Were there any pod crashes in the last 30 minutes?”

* “Why did that Job fail this morning?”

These natural-language prompts get translated into real queries and results. The outcome? Faster issue resolution, increased system transparency, and lower barrier to operational insights for both technical and non-technical users.

That's an example of the power of agentic AI.

== Meet Your New Toolkit

In this module, we'll use a preconfigured stack that includes:

* **Gen AI Studio Playground** - an integrated UI feature in Red Hat AI for testing model responses, MCP server integrations, and tool invocations before committing to production code. Use it to validate that models, prompts, and external connections work together correctly, then export the validated configuration to your application.

* **Kubernetes MCP Server**  - allows our LLM agent to interact with our live OpenShift cluster and deployed resources.

* **Rocket.Chat MCP Server** - allows our LLM agent to interact with a Rocket.Chat workspace. We will use it to interact with a specific Rocket.Chat workspace for which we have authorization configured.

* **NVIDIA Nemotron 3 Nano 30B A3B FP8 Model** - a lightweight, instruction-tuned model optimized for agentic workflows and tool calling. This model drives the agent's decision-making as it translates natural language into MCP server tool invocations.

IMPORTANT: This agent will have real access to your OpenShift cluster and Rocket.Chat workspace. When experimenting:

* Avoid deleting resources unless explicitly instructed
* Do not modify resources outside the namespaces specified in exercises
* Be cautious when sharing logs - they may contain sensitive information
* Always validate tool outputs before acting on them in production scenarios

=== Using the playground

This is where the magic happens. The playground lets you test-drive your entire agentic AI stack - model, prompts, MCP servers, and tool calls - **before writing a single line of application code**.

Think of it as your **sandbox for operational automation**. Instead of deploying blind and debugging in production, you validate the whole chain here:

* Does the model understand my prompts correctly?
* Are the MCP servers responding with the right data?
* Can I get pod logs, list namespaces, or post to chat channels reliably?

**The workflow:** Experiment here → validate behavior → export configuration → integrate into automation.

In the playground, you'll see a chat interface with your model preselected and a default system prompt. You can adjust the system prompt or temperature as needed.

NOTE: The **temperature** setting controls model creativity. Keep it **low** (0.0-0.3) for deterministic, accurate responses - critical when invoking tools or querying infrastructure. Higher temperatures introduce randomness, which can break tool calls.

image::playground.png[]

NOTE: Authorization to the MCP servers is **session-based**. If you refresh or reopen the playground, you'll need to re-select the MCP servers and reconfigure your settings.

**View Tools**

View the MCP servers' available tools. You can see the RocketChat MCP Server has 28 tools active and available:

image::view-rc-tools.png[]

NOTE: You will see an alert about **performance degradation with more than 40 active tools**. This happens because too many tools increase the model's context window size and reduce its ability to accurately select the correct tool for each task - similar to decision paralysis when faced with too many choices. The model must evaluate every available tool against each request, and more options mean more opportunity for selection errors.

To optimize performance and help the model focus on the most relevant operations, we'll deselect some of the less frequently used tools. This improves response accuracy and reduces latency for tool invocation.

* Select the Kubernetes MCP server tools:

image::view-k8s-tools.png[]

Ensure only the following tools are selected:

* `namespaces_list`
* `pods_get`
* `pods_list`
* `pods_list_in_namespace`
* `pods_log`

* Now select the Rocket.Chat MCP Server tools:

image::view-rc-tools.png[]

Ensure only the following tools are selected:

* `get_channel_info`
* `list_channels`
* `send_message`

NOTE: Feel free to experiment with other tools! For the purposes of the following steps, the above configuration is ideal to ensure consistent behavior.

== Investigate our OpenShift Resources

The tool information will give you guidance into how to interact with the model in chat to activate the tool calls correctly. You will need to utilize the right keywords so that you're engaging with the appropriate tools.

In the chat, enter:

[source,console,role=execute,subs=attributes+]
----
List all pods in the grafana namespace.
----

Response output will vary slightly, but you should see it correctly list the active pod in the `grafana` namespace. The pod name will start with `grafana-deployment-` followed by a generated suffix (e.g., `grafana-deployment-7d8f9c5b4-xyz12`).

✓ **Success indicator**: You should see output listing one or more pods with names starting with `grafana-deployment-` and showing their status as `Running`.

Let's try something else:

[source,console,role=execute,subs=attributes+]
----
Get logs for the grafana-deployment pod in the grafana namespace.
----

NOTE: You may need to use the exact pod name from the previous command (including the generated suffix). If the model cannot find the pod, copy the full pod name from the previous output and specify it in your request.

✓ **Success indicator**: You should see log output including timestamps. The logs will show the application's runtime activity.

Feel free to experiment further with the tools available before moving to the next step.

=== Post a Message to our Rocket.Chat Workspace

With our Rocket.Chat MCP Server, we can extend our agentic AI experience beyond Kubernetes and into team collaboration tools (among many other possibilities!).

This MCP server bridges your NVIDIA Nemotron model with a Rocket.Chat workspace to fetch and post approved data.

Open the Rocket.Chat application at: {rocketchat_url}

Login with the following credentials:

* **Username**: {user}
* **Password**: {password}

You will see a pop-up. Simply close it, or accept update configuration:

image::rocketchat-popup.png[]

NOTE: The Rocket.Chat workspace is pre-configured with a `#workshop` channel where your AI agents will post messages. You will observe the messages there after sending them through the playground.

**Why this matters:**

* SREs and DevOps teams often work across multiple collaboration channels.

* By giving your AI visibility into a collaboration space like Rocket.Chat (similar to Slack), you can use natural language to check team communication spaces or to post information without switching tools.

* This is a safe, controlled example — you can read channels and post messages through your authorized bot user.

==== In the Gen AI Studio Playground chat interface, type:

[source,console,role=execute,subs=attributes+]
----
List all Rocket.Chat channels in our workspace.
----

✓ **Success indicator**: You should see a list of available channels including `#general` and `#workshop`.

Now, let's post a message to our Rocket.Chat workspace:

[source,console,role=execute,subs=attributes+]
----
Post a message to the #workshop channel: "Hi from <insert name>".
----

Go check the `#workshop` channel in the Rocket.Chat application to see your message.

✓ **Success indicator**: Your message should appear in the #workshop channel within a few seconds, posted by your bot user account.

==== Send logs to Rocket.Chat

Now, let's try out a real use case for this! It may not be done through a chat UI like the playground, but it's a good example of how you can use agentic AI to help you with your work. In a production environment, you would likely use a more robust automation to send logs or other information from your Red Hat AI cluster to Rocket.Chat.

Let's send pod logs from your workspace to the `#workshop` channel, where you and your team can view them in the Rocket.Chat interface.

[source,console,role=execute,subs=attributes+]
----
Get the pod logs for the grafana-deployment pod in the grafana namespace. Review the logs for any errors or warnings, then post a summary highlighting any issues found along with the recent log entries to the #workshop channel in Rocket.Chat.
----

✓ **Success indicator**: Check the #workshop channel in Rocket.Chat - you should see a message containing both a summary of the log analysis and relevant log excerpts from the Grafana pod.

== Summary: What You Did

In this module, you:

* **Integrated your own LLM with a tool-using agent** - connecting NVIDIA Nemotron with MCP servers for live infrastructure interaction
* **Explored OpenShift resources with natural language** - querying pods, namespaces, and logs without writing kubectl commands
* **Interacted with a Rocket.Chat workspace using natural language** - posting operational data directly to team collaboration channels

**Why This Matters:**

This agentic pattern enables on-call engineers and SREs to query infrastructure using conversational language, reducing context switching and accelerating incident response. You've experienced the foundation of ChatOps workflows where AI agents bridge operational tools and team communication platforms.

**Real-World Applications:**

* **Automated incident response**: Query cluster health, retrieve logs, and post findings to Slack/Teams without leaving your chat window
* **Self-service infrastructure queries**: Enable non-technical team members to ask "Is the API healthy?" and get real answers
* **Proactive monitoring**: Agents can watch for specific patterns and automatically notify teams when issues arise

You just used AI to reduce operational complexity and speed up workflows - imagine scaling this across your entire infrastructure stack!
